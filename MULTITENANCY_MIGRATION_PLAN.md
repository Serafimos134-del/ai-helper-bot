# План миграции ai-helper-bot: один пользователь → продукт для трейдеров по подписке

Статус: план, утверждён к исполнению. Цель — не переписать проект, а
эволюционировать существующую архитектуру AI Trading Core в
мультипользовательский режим. Оплата — Crypto Pay (@CryptoBot API).

---

## Аудит: готовность к мультитенантности

### Хорошая новость: слой БД уже спроектирован под multi-tenancy

Почти все методы `services/database.py` уже принимают `user_id` параметром
(`get_closed_trades(user_id=...)`, `get_open_trades(user_id=...)`,
`add_behavior_event(user_id, ...)`, `memory_set/get(..., user_id=...)`,
`get_trade_events(order_id, user_id=...)`) — таблицы `open_trades`,
`closed_trades`, `behavior_events`, `trade_events`, `trader_memory` уже
имеют колонку `user_id`. Это не пришлось проектировать заново — только
перестать хардкодить `'default'` при вызовах.

`ai/trader_context.py`, `services/performance_engine.py`,
`services/behavior_engine.py` — тоже уже принимают `user_id`. AI Trading
Core (ConsensusEngine/JudgeAgent/TraderContext/DNA) не потребует
структурных изменений — только правильный `user_id` сверху вместо
`'default'`.

`users` table уже содержит `bingx_api_key`/`bingx_secret_key`
(шифруются через `services/crypto_utils.py`, Fernet, ключ из
`DB_ENCRYPTION_KEY` — реализация production-grade, уже готова),
`subscription_tier`, `subscription_expires_at`. Хранилище для ключей и
подписки **уже спроектировано** — просто не используется.

### Плохая новость: три места полностью single-user

**1. `services/bingx_api.py` — единственная настоящая блокирующая точка.**
Все функции (`get_balance`, `get_open_positions`, `get_kline`, ...)
подписывают запросы одной глобальной парой `BINGX_API_KEY`/
`BINGX_SECRET_KEY` из `.env` (`_sign()`, модульный уровень). Ни один
вызов не параметризован по пользователю. Пока это не исправлено — даже
идеальный онбординг ключей бесполезен: все запросы всё равно уйдут под
одним аккаунтом.

**2. Роутинг Telegram-апдейтов — захардкожен на `TELEGRAM_CHAT_ID`:**
- `bot.py`: `error_handler()` (строка 48), `restore_history_command`
- `handlers/menu.py`: `menu_handler()` — единственная точка входа для всех
  кнопок меню
- `handlers/system.py`: `_check_chat()` — гейт всех admin/system команд,
  плюс `sync_command`/`status_command` напрямую используют `CHAT_ID`
- `core/scheduler.py`: `setup_scheduler(app, db, CHAT_ID)` — один chat_id
  на весь процесс

**3. Фоновые джобы — обрабатывают одного пользователя:**
`services/auto_sync.py:219` — `user_id = 'default'` захардкожен
буквально в коде синхронизации. `core/scheduler.py`: `auto_sync_job`/
`position_watch_job` принимают один `chat_id`, не список пользователей.

### Что нужно привязать к user_id

Только `services/bingx_api.py` требует структурного изменения (ключи
параметром в каждую функцию вместо глобальных). Всё остальное уже
принимает `user_id` — требуется правильно его прокидывать, не
переписывать логику.

### Какие таблицы нужно расширить

`users` — добавить (safe-миграция, тот же паттерн `ALTER TABLE`, что уже
используется):
- `bingx_keys_valid` (BOOLEAN) — прошли ли ключи валидацию
- `bingx_keys_validated_at` (TIMESTAMP)
- `notifications_enabled` (BOOLEAN DEFAULT 1)
- `last_notification_sent_at` (TIMESTAMP) — защита от дублей при рестарте

Новая таблица `payments` — инвойсы Crypto Pay (`invoice_id`, `user_id`,
`amount`, `asset`, `status`, `created_at`, `paid_at`) — нужна для
сверки статуса оплаты и защиты от повторной обработки одного платежа
(идемпотентность вебхука/поллинга).

Остальные таблицы менять не нужно — уже готовы.

### Как хранить BingX-ключи пользователей

Механизм уже есть и не требует новой разработки: `set_bingx_keys()`/
`get_bingx_keys()` в `database.py`, шифрование Fernet. Нужно только: (1)
диалог ввода в Telegram, (2) валидация тестовым read-only запросом перед
сохранением, (3) реальное чтение этих ключей в `bingx_api.py` вместо
глобальных.

### Как реализовать подписку (Crypto Pay)

`@CryptoBot` (Crypto Pay API) — создание инвойса через REST
(`createInvoice`), получение статуса (`getInvoices` поллингом или
webhook на `invoice_paid`). При успешной оплате — `subscription_expires_at
= max(now, текущий expires_at) + 14 дней` (продление добавляет к
оставшемуся сроку, а не от текущего момента — иначе повторная покупка до
истечения "сжигает" остаток).

**Блокер**: нужен API-токен приложения Crypto Pay (заводится в
`@CryptoBot` → Crypto Pay → Create App). Без него подключить оплату
невозможно даже в тестовом режиме — это первое, что нужно от вас, когда
дойдём до Этапа 4.

### Разграничение данных между пользователями

Раз БД-слой уже `user_id`-aware — дисциплина простая: **везде передавать
`str(update.effective_user.id)` вместо `'default'`**, ни одного
хардкода не должно остаться на путях, которые обрабатывают реальных
пользователей. Отдельно проверить: `core/container.py` — синглтоны
`ConsensusEngine`/`AIOrchestrator` не хранят пользовательское
состояние (BingX-ключи должны идти параметром на каждый вызов, не через
конструктор) — структура уже это позволяет, ничего менять не нужно.

## Критично для первого публичного запуска

1. **Мультитенантность BingX** — `bingx_api.py`: ключи параметром в
   каждую функцию вместо глобальных `.env`
2. **Хранение пользователей** — уже есть, только начать реально
   использовать `get_or_create_user()` на входе в каждый хендлер
3. **Хранение API-ключей** — уже есть (`set_bingx_keys`/шифрование),
   нужен диалог ввода + валидация
4. **Убрать единый `TELEGRAM_CHAT_ID`-гейт** — заменить на per-user
   проверку (подписка активна?) во всех хендлерах
5. **Подписка (Crypto Pay)** — инвойс, проверка оплаты, продление,
   таблица `payments`
6. **Гейт по истечении подписки** — блокировка команд + экран продления
7. **Мультипользовательские фоновые джобы** — `auto_sync_job`/
   `position_watch_job` — цикл по `get_all_active_users()`, не один
   `chat_id`
8. **Ежедневные уведомления** — простой daily-джоб (напоминание/дайджест),
   с защитой от дублей (`last_notification_sent_at`)
9. **Персональная аналитика** — уже реализовано (TraderContext/DNA/
   consilium) — становится "просто работает" как только пункты 1-4
   решены, отдельной разработки не требует

## Порядок выполнения (зависимости определяют порядок, не приоритет "на глаз")

```
Этап 1: bingx_api.py — ключи параметром           [фундамент, ничего не работает без этого]
Этап 2: онбординг BingX-ключей + валидация         [зависит от Этапа 1]
Этап 3: убрать CHAT_ID-гейт, per-user роутинг      [независимо от 1-2, можно параллельно]
Этап 4: Crypto Pay — подписка/оплата/продление     [нужен токен от пользователя]
Этап 5: гейт по истечении подписки                 [зависит от Этапа 4]
Этап 6: мультипользовательские фоновые джобы        [зависит от 1+3]
Этап 7: ежедневные уведомления                     [зависит от 6]
Этап 8: сквозной прогон полного пути                [зависит от всего]
```

Этапы 1 и 3 не зависят друг от друга — начинаю с Этапа 1 (BingX-ключи),
поскольку это самое инвазивное и рискованное изменение (легко перепутать
чьи-то ключи между пользователями при ошибке) — лучше сделать и
протестировать его первым, пока внимания больше всего.
