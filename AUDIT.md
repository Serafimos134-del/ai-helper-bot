# Аудит проекта ai-helper-bot

Дата: 2026-07-05

## Итог

Это бот-советник/журнал сделок (BingX API), а не бот, который сам торгует — в `services/bingx_api.py` нет эндпоинтов на открытие/изменение ордеров, только чтение баланса, позиций и рыночных данных. Это снижает риск прямой потери денег из-за багов, но AI-рассчитанные риск/SL/TP всё равно показываются человеку, который может действовать по ним — так что ошибки в расчётах остаются серьёзными.

Три самых срочных проблемы:

1. `webhook_server.py` делает `git reset --hard` + рестарт сервиса по любому push с валидной подписью `WEBHOOK_SECRET` — утечка одного секрета даёт полный удалённый доступ к серверу (RCE).
2. `services/calc_engine.py` дважды применяет плечо при расчёте размера позиции — реальный риск может оказаться в 10–100 раз больше заявленного процента.
3. API-ключи BingX хранятся в SQLite в открытом виде, без шифрования.

В целом качество кода выше среднего для такого проекта: единообразное логирование, retry/timeout на сетевых вызовах, WAL-режим SQLite с блокировками, продуманный "деградированный" режим в AI consensus engine. Но есть реальные, непроверенные арифметические баги в торговых движках — их стоит исправить, прежде чем доверять цифрам.

## 1. Безопасность

| Файл:строка | Критичность | Проблема | Как исправить |
|---|---|---|---|
| `webhook_server.py:11,48-51` | Критично | Эндпоинт деплоя выполняет `git fetch && git reset --hard origin/main` и `systemctl restart` при любом push с валидной HMAC-подписью `WEBHOOK_SECRET`. Нет IP-allowlist, нет rate limiting. Утечка секрета = RCE на сервере. | Ограничить доступ по IP GitHub, запускать сервис от непривилегированного пользователя, ротировать секрет. |

## 2. Финансовая/торговая логика

| Файл:строка | Критичность | Проблема | Как исправить |
|---|---|---|---|
| `services/calc_engine.py:19-21` | Критично | `position_size = margin * leverage / price`, где `margin` уже включает `leverage` — плечо применяется дважды. При balance=$1000, risk=1%, leverage=10x реальный размер позиции завышен в leverage раз (100x риска вместо 10x). | Считать `position_size = risk_amount / abs(entry - stop_loss)` (риск-ориентированный расчёт) либо применять leverage один раз. |
| `calc_engine.py:30-31` vs `ai/risk_engine.py:59-63` | Средняя | Две разные формулы цены ликвидации в разных модулях — дадут разные числа для одной и той же позиции. | Свести в одну общую протестированную функцию, сверить с документацией BingX. |
| `ai/engines/scoring_engine.py:69-82` vs `judge_agent.py:53-58` | Высокая | `_calc_risk()` возвращает "чем выше — тем опаснее", но `JudgeAgent` складывает его с положительным весом, как будто выше = лучше. Опасная позиция без SL/TP и с высоким плечом может завышать итоговый score вместо понижения. | Инвертировать шкалу перед передачей в judge, либо изменить знак веса. |
| `services/stop_engine.py:43,46` | Средняя | Дефолтный SL — плоские 5% от цены входа, без учёта волатильности актива. | Масштабировать по ATR/классу волатильности. |
| `services/tp_engine.py:59-71` | Низкая | Аналогично — фиксированные % для TP без учёта волатильности. | Тот же fix. |
| `services/auto_sync.py:62-69` | Средняя | Цена выхода оценивается как `entry + pnl/qty` на основе последнего known unrealized PnL, а не реальной цены закрытия — это оценка, не факт. | Помечать в сообщениях как "оценочно", либо сверять с `get_closed_orders()`. |
| `database.py` / `bot.py:70` / `migrate_to_sqlite.py:7-15` | Низкая | Маппинг BUY/SELL→LONG/SHORT продублирован в разных местах, в `migrate_to_sqlite.py` есть недостижимая ветка кода. | Вынести в общий `ai/engines/normalizer.py`, использовать везде. |
| `ai/risk_engine.py:77-99` | Низкая | Расчёт риска портфеля не делится на leverage — по сути это риск в дистанции цены, не в марже. Не документировано как упрощение. | Задокументировать или уточнить с учётом margin mode. |
| `database.py` (много мест) | Низкая | Мультипользовательская схема (таблица `users`, ключи на пользователя) есть, но почти весь код хардкодит `user_id='default'`. | Либо довести мультитенантность до конца с тестами, либо убрать заготовку. |

## 5. Архитектура

- `core/container.py` — простой service locator (не настоящий DI), но рабочий для масштаба проекта.
- `core/router.py` — маршрутизация callback-кнопок через цепочку `if/elif`; читаемо для ~10 типов, но содержит бизнес-логику (`generate_full_ai_analysis`), которая дублирует то, что уже делает `auto_sync.py:_analyze_and_notify`.
- Разделение `handlers/` (Telegram-специфика) и `services/`+`ai/` (бизнес-логика) в целом выдержано хорошо.
- Пакет `ai/` организован чисто: `agents/` / `engines/` / `providers/` — разумное разделение. `ConsensusEngine` с таймаутами по агентам, деградированным режимом и реальной статистической метрикой разногласия — качественное решение, одна из сильных сторон проекта (омрачается багом со знаком risk_score, см. раздел 2).
- `webhook_server.py` архитектурно не связан с остальным ботом — отдельное Flask-приложение для деплоя. Учитывая его высокие привилегии (`git reset --hard` + рестарт), логичнее вынести в отдельный ops-репозиторий.

## 6. Зависимости

| Пакет | Версия | Комментарий |
|---|---|---|
| `python-telegram-bot[job-queue]` | `==20.7` | Релиз конца 2023 — на середину 2026 заметно устарел, стоит проверить актуальные советы по безопасности и обновиться. |
| `httpx[socks]` | `==0.27.0` | Начало 2024, есть более новые релизы — проверить advisories. |
| `requests[socks]` | `==2.31.0` | Версия-фикс для CVE-2023-32681 (утечка `.netrc` при редиректе) — сам фикс уже включён, но стоит проверить более поздние CVE. |
| `flask` | `==3.0.0` | Самый чувствительный пакет с точки зрения безопасности, т.к. `webhook_server.py` смотрит наружу — проверить advisories Flask/Werkzeug после 3.0.0. |
| `python-dotenv` | `==1.0.0` | Риск минимален. |
| `groq` | `>=0.9.0` | Нижняя граница без верхней — риск непредсказуемых обновлений. |
| `aiohttp` | `>=3.9.0` | Та же проблема; у aiohttp исторически были CVE (например, request smuggling) — стоит зафиксировать точную версию. |

**Рекомендация**: зафиксировать точные версии всех пакетов (заменить `>=` на `==`) и прогнать `pip-audit`/`safety check` — этот аудит не может достоверно подтвердить текущий статус CVE без прямого запроса к базам PyPI/OSV.
