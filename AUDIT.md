# Аудит проекта ai-helper-bot

Дата: 2026-07-05

## Итог

Это бот-советник/журнал сделок (BingX API), а не бот, который сам торгует — в `services/bingx_api.py` нет эндпоинтов на открытие/изменение ордеров, только чтение баланса, позиций и рыночных данных. Это снижает риск прямой потери денег из-за багов, но AI-рассчитанные риск/SL/TP всё равно показываются человеку, который может действовать по ним — так что ошибки в расчётах остаются серьёзными.

Три самых срочных проблемы:

1. `webhook_server.py` делает `git reset --hard` + рестарт сервиса по любому push с валидной подписью `WEBHOOK_SECRET` — утечка одного секрета даёт полный удалённый доступ к серверу (RCE).
2. `services/calc_engine.py` дважды применяет плечо при расчёте размера позиции — реальный риск может оказаться в 10–100 раз больше заявленного процента.
3. API-ключи BingX хранятся в SQLite в открытом виде, без шифрования.

В целом качество кода выше среднего для такого проекта: единообразное логирование, retry/timeout на сетевых вызовах, WAL-режим SQLite с блокировками, продуманный "деградированный" режим в AI consensus engine. Но есть реальные, непроверенные арифметические баги в торговых движках — их стоит исправить, прежде чем доверять цифрам.

## 2. Финансовая/торговая логика

| Файл:строка | Критичность | Проблема | Как исправить |
|---|---|---|---|
| `services/calc_engine.py:19-21` | Критично | `position_size = margin * leverage / price`, где `margin` уже включает `leverage` — плечо применяется дважды. При balance=$1000, risk=1%, leverage=10x реальный размер позиции завышен в leverage раз (100x риска вместо 10x). | Считать `position_size = risk_amount / abs(entry - stop_loss)` (риск-ориентированный расчёт) либо применять leverage один раз. |
| `calc_engine.py:30-31` vs `ai/risk_engine.py:59-63` | Средняя | Две разные формулы цены ликвидации в разных модулях — дадут разные числа для одной и той же позиции. | Свести в одну общую протестированную функцию, сверить с документацией BingX. |
| `ai/engines/scoring_engine.py:69-82` vs `judge_agent.py:53-58` | Высокая | `_calc_risk()` возвращает "чем выше — тем опаснее", но `JudgeAgent` складывает его с положительным весом, как будто выше = лучше. Опасная позиция без SL/TP и с высоким плечом может завышать итоговый score вместо понижения. | Инвертировать шкалу перед передачей в judge, либо изменить знак веса. |
| `services/stop_engine.py:43,46` | Средняя | Дефолтный SL — плоские 5% от цены входа, без учёта волатильности актива. | Масштабировать по ATR/классу волатильности. |
| `services/tp_engine.py:59-71` | Низкая | Аналогично — фиксированные % для TP без учёта волатильности. | Тот же fix. |
| `services/auto_sync.py:62-69` | Средняя | Цена выхода оценивается как `entry + pnl/qty` на основе последнего known unrealized PnL, а не реальной цены закрытия — это оценка, не факт. | Помечать в сообщениях как "оценочно", либо сверять с `get_closed_orders()`. |
| `database.py` / `bot.py:70` / `migrate_to_sqlite.py:7-15` | Низкая | Маппинг BUY/SELL→LONG/SHORT продублирован в разных местах, в `migrate_to_sqlite.py` есть недостижимая ветка кода. | Вынести в общий `ai/engines/normalizer.py`, использовать везде. |
| `ai/risk_engine.py:77-99` | Низкая | Расчёт риска портфеля не делится на leverage — по сути это риск в дистанции цены, не в марже. Не документировано как упрощение. | Задокументировать или уточнить с учётом margin mode. |
| `database.py` (много мест) | Низкая | Мультипользовательская схема (таблица `users`, ключи на пользователя) есть, но почти весь код хардкодит `user_id='default'`. | Либо довести мультитенантность до конца с тестами, либо убрать заготовку. |

## 5. Архитектура

- `core/container.py` — простой service locator (не настоящий DI), но рабочий для масштаба проекта.
- `core/router.py` — маршрутизация callback-кнопок через цепочку `if/elif`; читаемо для ~10 типов, но содержит бизнес-логику (`generate_full_ai_analysis`), которая дублирует то, что уже делает `auto_sync.py:_analyze_and_notify`.
- Разделение `handlers/` (Telegram-специфика) и `services/`+`ai/` (бизнес-логика) в целом выдержано хорошо.
- Пакет `ai/` организован чисто: `agents/` / `engines/` / `providers/` — разумное разделение. `ConsensusEngine` с таймаутами по агентам, деградированным режимом и реальной статистической метрикой разногласия — качественное решение, одна из сильных сторон проекта (омрачается багом со знаком risk_score, см. раздел 2).
- `webhook_server.py` архитектурно не связан с остальным ботом — отдельное Flask-приложение для деплоя. Учитывая его высокие привилегии (`git reset --hard` + рестарт), логичнее вынести в отдельный ops-репозиторий.

Добавлен `ai/orchestrator.py` — класс `AIOrchestrator`, единая точка входа в
AI Trading Core, зарегистрирован в `core/container.py` как `get_orchestrator()`.
Все точки вызова (`handlers/ai.py`, `core/router.py`, `services/auto_sync.py`)
переведены с прямого обращения к `get_consensus()`/`ConsensusEngine` на
`get_orchestrator()`.

Маршрутизация запросов (согласно плану, Этап 3):

| Тип запроса | Агенты (по плану) | Метод |
|---|---|---|
| `open_position` | Position Analyst + Risk Manager + Judge | `review_open_position()` |
| `closed_trade` | Trade Reviewer + Risk Manager + Judge | `review_closed_trade()` |
| `new_setup` | Market Analyst + Strategy Advisor + Risk Manager + Judge | `evaluate_new_setup()` |

**Важное архитектурное решение, которое стоит зафиксировать:** роли Position
Analyst / Trade Reviewer / Strategy Advisor из плана **не выделены в отдельные
классы**. Они уже существуют как режимы (`mode='open'|'post_trade'|'setup'`)
внутри `MarketAgent`/`RiskAgent`/`PsychologyAgent`, которыми управляет
`ConsensusEngine` — именно он и делает основную работу по подбору поведения
под тип запроса. Выделение их в отдельные классы прямо сейчас было бы чистым
переименованием без изменения поведения (запрещённая шаблоном
преждевременная абстракция). `AIOrchestrator` — тонкий слой поверх
`ConsensusEngine`: документирует маршрутизацию из плана, даёт единую точку
входа и логирование по типу запроса, и место, куда позже можно подставить
принципиально новых агентов (Portfolio/News/Macro Analyst — Этап 3) не
меняя вызывающий код.

**Побочная находка — исправлено в этом же проходе.** При чтении
`ai/agents/risk_agent.py` и `ai/agents/psychology_agent.py` обнаружилось, что
их методы `_analyze_open_position`/`_analyze_setup`/`_analyze_post_trade`
возвращали `risk_score`/`psychology_score` в шкале 0–10, тогда как
`JudgeAgent` и `ScoringEngine` работают в шкале 0–100. На тот момент это не
было активным багом: `ConsensusEngine._run_agents_parallel()` не использовал
эти числовые поля для `open`/`post_trade`/`setup` — реальные
`risk_score`/`psychology_score`, которые видит Judge, считает отдельно
`ScoringEngine.calculate()`, а из ответов RiskAgent/PsychologyAgent берётся
только текстовое `summary`. Но это была скрытая ловушка: 0–10 значения
выглядели валидными и лежали прямо в JSON-ответе агента — если бы в будущем
код случайно начал брать `risk_score`/`psychology_score` напрямую из ответа
агента вместо `ScoringEngine`, Judge получил бы на порядок заниженные числа
и считал бы почти любую позицию критически рискованной.

Исправление:
- `ScoringEngine._calc_risk`/`_calc_psychology`/`_calc_data_quality` сделаны
  публичными (`calc_risk`/`calc_psychology`/`calc_data_quality`) — это теперь
  единственная реализация шкалы риска 0–100 (выше = безопаснее) в проекте.
- `RiskAgent._calc_risk_score()` (открытая позиция/закрытая сделка) больше не
  дублирует логику проверки SL/TP/leverage в своей шкале — делегирует в
  `ScoringEngine.calc_risk()`.
- `RiskAgent._analyze_setup()` — свой более богатый, направленно-зависимый
  расчёт риска сетапа сохранён (он не дублирует `ScoringEngine`, а несёт
  доп. сигнал), но приведён к шкале 0–100 с той же ориентацией "выше =
  безопаснее" (раньше была обратная ориентация: выше = опаснее).
- `PsychologyAgent._analyze_setup`/`_analyze_open_position`/`_analyze_post_trade`
  — логика не дублирует `ScoringEngine` (анализ формулировок, early-exit и
  т.п.), поэтому просто пересчитана из шкалы 0–10 в 0–100 с сохранением всех
  относительных весов (все базовые значения и штрафы ×10).
- Проверено смоук-тестом: безопасная позиция (SL+TP, leverage 3x) даёт
  risk_score=95/psychology_score=100, рискованная (без SL/TP, leverage 15x) —
  risk_score=50/psychology_score=30; сетап LONG+TRENDING_UP — risk_score=70,
  LONG+UNKNOWN regime — risk_score=20. Направление шкалы везде одинаковое.

## Этап 4 плана AI Trading Core: анализ открытых позиций (2026-07-09)

При подготовке к Этапу 4 выяснилось, что детерминированный движок для всех
пунктов плана ("актуален ли стоп", "перенести стоп", "частично зафиксировать

`services/stop_engine.py`, `services/tp_engine.py`, `services/ai_decision_engine.py`),
но не был подключён ни к одному живому потоку — единственный вызывающий код,
`handlers/system.py:analyze_command` (`/analyze`), нигде не зарегистрирован
как команда (см. докстринг `bot.py`: "removed /analyze and /calc from public
commands"). Это ровно та "Высокая (пока не активна)" находка про
`api_cache.set(ttl=...)` из раздела 3 исходного аудита — сам баг с `ttl`
давно исправлен, но команда так и осталась незарегистрированной.

Вместо регистрации отдельной `/analyze` (что вернуло бы фрагментацию на два
независимых способа анализа позиции, чего план явно избегает) —
`AIOrchestrator.review_open_position()` теперь дополнительно строит
`position_plan`, вызывая тот же `ai_decision_engine.analyze_decision()` на
основе живого `market_data.get_market_snapshot()`, и подмешивает его в общий
ответ. Таким образом один вызов `/consilium → Анализ позиции` теперь
закрывает весь список Этапа 4:

- качество входа и текущий риск — от AI-консилиума (Market/Risk/Psychology
  agents + Judge, как и раньше);
- актуальность стопа, рекомендация по переносу, частичная фиксация прибыли,
  решение HOLD/EXIT/DCA/PARTIAL_TP/FULL_TP — от `position_plan`.

Технические детали:
- `analyze_decision()`/`stop_engine`/`tp_engine` ожидают нормализованные
  snake_case поля (`entry_price`, `unrealized_pnl`, ...), а не сырой ответ
  BingX API (`entryPrice`, `unrealizedPnl`, ...) — `_build_position_plan()`
  прогоняет позицию через уже существующий `normalize_position()`.
- Позиции в `/consilium` берутся из `services.bingx_api.get_open_positions()`
  (живые данные биржи), а не из `db.get_open_trades()` — поэтому поля Trade
  Management Engine v2 (`dca_count`, `invalidation_sl`, `tp_zones`, `idea`,
  задаются через `/setidea`) у них отсутствуют. `_build_position_plan()`
  пробрасывает эти поля, если они есть на объекте, иначе `analyze_decision()`
  использует безопасные дефолты (без DCA-рекомендаций, инвалидация — по
  структуре рынка, а не ручной отметке). Это не регрессия — так же вело себя
  бы прямое использование `get_open_positions()` и раньше; полное объединение
  источников позиций (BingX API + DB) — отдельная, более крупная задача, не
  делалась в этом проходе.
- `handlers/system.py:analyze_command` (`/analyze`) намеренно не тронут и не
  удалён — теперь он избыточен (то же самое, но на других данных и без AI-
  консилиума), но его никто не просил менять. Кандидат на удаление в
  отдельном проходе по чистке кода.
- Проверено смоук-тестом на синтетических свечах с восходящим трендом:
  `position_plan` корректно строит `structure`/`stop`/`tp`/`decision`, а
  `handlers/ai.py:_build_response()` корректно рендерит их в ответ консилиума.

## Этап 6 плана AI Trading Core: анализ новых сетапов (2026-07-09)

Требование плана: по запросу вида "BTC Long" бот строит полный торговый план
(сценарий, аргументация, цена входа, Stop Loss, TP1-3, Risk/Reward, размер

недостающими цифрами вместо отдельного механизма) — `evaluate_new_setup()`
теперь дополнительно строит `trade_plan`:

- Гипотетическая позиция (`entry_price` = текущая рыночная цена из
  `market_data.get_market_snapshot()`, `side` = направление сетапа) прогоняется
  через те же `stop_engine.analyze_stop()`/`tp_engine.analyze_tp()`, что уже

  рынка и цены/стороны, так что дают структурные SL/TP1/TP2/TP3(runner) и
  для ещё не открытой сделки.
- Risk/Reward считается как `|TP1 - entry| / |entry - SL|`.
- Размер позиции — через уже существующий `services/calc_engine.calculate_position()`
  (тот же расчёт, что стоит за незарегистрированной командой `/calc`, см.
  находку Этапа 4 про `/analyze` — здесь та же картина: рабочий калькулятор,
  просто не подключённый к пользовательскому потоку). Параметры взяты из уже
  задокументированных пользователю правил бота (закреплённый статус,
  `core/scheduler.py:_build_status_text`, раздел "Правила"): вход 10% от
  депо (`risk_percent=10`), плечо x5 для BTC/ETH и x3 для остальных пар
  (`utils.liquidation.get_volatility_class`), маржа Cross — а не
  собственные дефолты `/calc` (`risk_percent=1.0`, isolated), которые
  предназначены для произвольного ручного расчёта, а не для правил именно
  этого бота.
- `handlers/ai.py:_build_response()` рендерит `trade_plan` (вход/SL/TP1-3/RR/
  размер позиции) в ответе консилиума по новому сетапу.

Проверено смоук-тестом на синтетических свечах с восходящим трендом и
замоканным балансом ($1000 equity): `trade_plan` для BTC LONG даёт вход,
SL, TP1-3, RR, leverage=5x (BTC — низкая волатильность), размер позиции и
маржу; `_build_response()` корректно рендерит план в текст ответа.

## Баги из отчёта пользователя (2026-07-09), вне плана по этапам

Пользователь сообщил о двух проблемах при реальном использовании бота.
Разбор показал три отдельные причины (первая — общая для обоих симптомов):

1. **`ReplyKeyboardMarkup`/`ForceReply` передавался в `edit_message_text`.**
   Telegram Bot API у `editMessageText` принимает только
   `InlineKeyboardMarkup` в `reply_markup` — попытка отредактировать
   сообщение с `cancel_keyboard()` (обычная нижняя клавиатура) в ветках
   `comment_`/`entry_reason_`/`exit_reason_` (`core/router.py`) роняла весь
   вызов ошибкой Telegram API. Проявлялось как "кнопка не реагирует":
   пользователь нажимал "✏️ Комментарий" — сообщение не менялось, поле
   ввода не появлялось. Причина второй жалобы пользователя ("надо чтобы
   лезло окошко куда писать комментарий") — то же самое: окошко ввода
   физически не могло появиться, вызов падал раньше. Исправлено: сообщение
   с кнопками теперь редактируется без `reply_markup`, а поле ввода
   открывается отдельным новым сообщением с `ForceReply` (нативный
   Telegram-механизм — реально фокусирует клавиатуру на вводе текста,
   с плейсхолдером, в отличие от простой подсказки текстом).
2. **Ни один `elif`-блок в `core/router.py:handle_callback` не был обёрнут
   в try/except.** Любое исключение (включая баг #1, а также, например,
   `BadRequest` от Telegram при непарном `*`/`_` в пользовательском
   комментарии, попавшем в `parse_mode='Markdown'`) уходило только в
   глобальный `error_handler` (`bot.py`), который лишь логирует и не отвечает
   пользователю — кнопка выглядела так, будто "не нажимается". Исправлено:
   весь диспетчер обёрнут в try/except с сообщением об ошибке пользователю;
   заодно карточка деталей сделки (`detail_`) переведена на обычный текст без
   `parse_mode`, т.к. в неё подставляется свободный текст пользователя
   (`exit_comment`/`comment`), а `generate_ai_review()` теперь чистит LLM-текст
   через `clean_markdown()` (как и остальные места, рендерящие AI-ответы) —
   раньше он был единственным необработанным.
3. **`state == 'consilium_choose_position'` перехватывал кнопку "Назад" раньше,
   чем до неё доходила общая проверка `text == BTN_BACK`** (`handlers/menu.py`).
   Нажатие "Назад" на клавиатуре выбора позиции уходило в
   `consilium_analyze_position` как будто это название позиции; та не находила
   совпадения, просила выбрать позицию заново и не сбрасывала state — кнопка
   "Назад" переставала работать до перезапуска диалога с ботом. Исправлено:
   `BTN_BACK` в этом состоянии обрабатывается отдельно и сбрасывает state.

Дополнительно: раз кнопка "Отмена" для `entering_comment_inline`/
`entering_exit_reason`/`entering_entry_reason` больше не показывается как
кнопка (заменена на `ForceReply` без своей клавиатуры), отмена этих
состояний в `handlers/menu.py` теперь принимает и обычный набранный текст
"отмена"/"cancel", не только нажатие кнопки.

Проверено смоук-тестами: детали сделки с непарными `*`/`_` в комментарии
рендерятся без `parse_mode`; симулированное исключение внутри диспетчера
долетает до пользователя текстом ошибки вместо молчаливого падения;
`entry_reason_`/`exit_reason_`/`comment_` редактируют исходное сообщение без
`reply_markup` и открывают `ForceReply` новым сообщением; набор "отмена"
текстом сбрасывает state и очищает `user_data`; кнопка "Назад" на выборе
позиции сбрасывает `consilium_choose_position` и возвращает в меню консилиума.

  здесь есть полный контекст Trade Manager v2: `dca_count`,
  `invalidation_sl`, `tp_zones`). Уведомляет только когда `decision` +
  статус стопа изменились с прошлой проверки (дедупликация через
  `trader_memory`, категория `position_watch`, ключ — `orderId`) — иначе
  бот писал бы одно и то же на каждом тике. Первая проверка новой позиции
  с нейтральным состоянием (HOLD + стоп на уровне инвалидации) не
  уведомляет — иначе бот сразу после открытия сделки слал бы "решение:
  HOLD" как будто это новость.
- Частота проверки каждого символа масштабируется по volatility-классу
  (`utils.liquidation.get_volatility_class`) — идея из
  `TRADE_MANAGEMENT_V3.md`, п.1 Trade Classification: HIGH (мемкоины и
  подобное) проверяется на каждом тике, MEDIUM — раз в 2 тика, LOW
  (BTC/ETH) — раз в 3. `job_queue` не поддерживает разное расписание на
  символ, поэтому это "раз в N тиков" одного job, а не N отдельных задач.

Проверено смоук-тестом: позиция с состоянием HOLD/keep на первой проверке
не уведомляет; при смене на breakeven — уведомление отправляется с текстом
из `format_position_plan()`; повторная проверка того же состояния breakeven
дедуплицируется (уведомления нет). Volatility-gating проверен по всем
классам (`BTC-USDT`/`ETH-USDT` → 3 тика, `SOL-USDT` → 2, `DOGE-USDT`/
`PEPE-USDT` → 1).

## TraderContext: advisory-интеграция в JudgeAgent (2026-07-09)

По итогам `TRADER_INTELLIGENCE_ARCHITECTURE.md` — первая часть плана
консолидации (§8, Этапы 2-4: единый `TraderContext`, интеграция в AI
Orchestrator, интеграция в JudgeAgent), без пятого пункта (удаление
дублей — отложено до проверки влияния на реальных решениях).

### Что сделано

- `ai/trader_context.py` — новый модуль, **без новых таблиц**:
  - `build_trader_context(db, symbol=None)` — runtime-объект (`profile`/
    `edges`/`streaks`/`behavior`/`symbol_context`), собирается из уже
    существующих `PerformanceEngine.get_full_report()` и
    `db.get_recent_behavior_events()` (последний — та самая находка из
    предыдущего аудита: метод существовал, но ни разу не вызывался нигде
    в проекте; теперь используется впервые).
  - `compute_dna_adjustment(trader_context)` — **advisory-only** поправка:
    возвращает `active=False` (поправка не применяется), если сделок
    меньше `MIN_TOTAL_TRADES=5`; поправка по конкретному символу требует
    отдельно `MIN_SYMBOL_TRADES=3` по этому символу; итоговая поправка
    всегда обрезается по модулю до `MAX_SCORE_ADJUSTMENT=10` — не может
    развернуть решение с ENTER(85+) в AVOID(<55) в одиночку.
- `ai/context_builder.py`: `trader_context` добавлен во все три метода
  сборки контекста (`build_for_open_position`/`build_for_new_setup`/
  `build_for_closed_trade`) — собирается параллельно с остальным
  контекстом через `asyncio.gather`, не блокирует существующие вызовы.
  Старые `history`/`memory`/`trader_profile`-заглушки **не удалены** — по
  явному решению владельца продукта удаление дублей идёт последним шагом,
  после проверки, что новый канал реально работает.
- `ai/consensus_engine.py`: `context.get('trader_context')` прокидывается
  в `self.judge.synthesize(...)`.
- `ai/agents/judge_agent.py`: `synthesize()` получил новый необязательный
  параметр `trader_context=None` (обратная совместимость — старые вызовы
  без этого параметра ведут себя как раньше). Поправка применяется к
  `final_score` **до** расчёта `verdict` (не косметика после), результат
  включает `base_score` (для сравнения) и `dna_adjustment` (для
  прозрачности — видно, откуда взялась поправка), причина поправки
  добавляется в `warnings`.

### Демонстрация: как меняется решение (одинаковые Market/Risk/Psychology
входы во всех сценариях — единственная переменная — `TraderContext`)

Синтетический трейдер: 6 сделок по `PEPE-USDT` (1 победа, winrate 17%,
недавний флаг `revenge_trading`), 4 сделки по `BTC-USDT` (3 победы, winrate
75%), общий winrate 40%.

| Сценарий | final_score | base_score | verdict | Поправка |
|---|---|---|---|---|
| Без `TraderContext` (baseline) | 78 | 78 | **ENTER** | — (`active=False`) |
| Новый сетап по PEPE-USDT (слабый символ + revenge flag) | **68** | 78 | **WAIT** | −10: "winrate 17% против среднего 40%; недавние признаки revenge trading" |
| Новый сетап по BTC-USDT (сильный символ) | 77 | 78 | ENTER (не изменился) | −1: символ сильный, поправка почти нулевая несмотря на глобальный флаг |
| Трейдер с 1 сделкой (недостаточно данных) | 78 | 78 | ENTER | — (`active=False`, ограничитель `MIN_TOTAL_TRADES` сработал) |

Ключевой результат: **на слабом для трейдера символе вердикт реально
меняется (ENTER → WAIT)**, на сильном символе почти не меняется (−1 из
78), при недостатке данных поправка не включается вовсе. Ограничители
влияния работают именно так, как требовалось ("advisory, не
переворачивает решение").

Полная интеграционная проверка (не только `JudgeAgent` в изоляции):
`ConsensusEngine.analyze_new_setup()` с замоканными сетевыми вызовами
BingX и замоканными `MarketAgent`/`RiskAgent`/`PsychologyAgent.analyze()`
— подтверждено, что `trader_context` доходит и до промпта `MarketAgent`
(видно в перехваченном `ctx` внутри мока), и до `JudgeAgent.synthesize()`
через реальный `context_builder.py` → `ConsensusEngine` путь, без ручной
подмены на уровне вызова Judge.

### Не сделано в этом проходе (сознательно, по плану)

- `MarketAgent`/`RiskAgent`/`PsychologyAgent`'s LLM-промпты **не изменены**
  для явного текстового упоминания `trader_context` (только сырые данные

- `MemoryEngine`/`_calculate_behavior_metrics()`/старые `trader_profile`-

## Trader DNA v1: реализация каталога паттернов + DNA Score (2026-07-09)

По итогам `TRADER_DNA_V1.md` — реализована первая часть роадмапа ("DNA v1"),
строго в его границах: новый read-only модуль, ни новых таблиц, ни новых
агентов, Vision и Premium AI не тронуты.

**Новый файл `ai/trader_dna.py`** — детерминированные функции поверх уже
существующих `closed_trades`/`behavior_events`/`PerformanceEngine`. Отдельный
модуль от `ai/trader_context.py`: `trader_context.py` — то, что видит
`JudgeAgent` (advisory-поправка к оценке сделки), `trader_dna.py` — витрина
для пользователя (каталог паттернов + DNA Score), не влияет на решения
консилиума.

Фаза 2 (каталог паттернов, только пункты ✅ из документа):
- `market_regime_pattern()` — результат по `market_trend` (тривиальный
  `GROUP BY` по уже существующей колонке, ранее нигде не считался).
- `holding_time_pattern()` — переудержание убыточных позиций (сравнение
  средней `holding_minutes` в победах/убытках).
- `early_exit_pattern()` — ранняя фиксация прибыли относительно `take_profit`.
- `stop_width_pattern()` — отклонение дистанции стопа от собственной средней
  нормы трейдера (не внешняя "правильность" — её взять неоткуда).
- `quality_after_loss_streak_pattern()` — падает ли `ai_score` сразу после
  серии убытков относительно базового среднего.
- `sl_tp_discipline()` — доля сделок с одновременно выставленными SL и TP.
- `behavior_symbol_pattern()` + `build_pattern_insights()` — приближённый
  джойн конкретного поведенческого флага (`fomo`/`revenge_trading`/
  `panic_close`) к сделкам того же символа по ближайшему времени (окно
  `BEHAVIOR_JOIN_WINDOW_MINUTES=15`, так как `behavior_events` не хранит
  `order_id` — см. `TRADER_DNA_V1.md` §1.1); при >1 совпадении по времени
  событие не засчитывается вместо угадывания. `build_pattern_insights()` —
  ровно та функция, что была намечена документом (Фаза 2, «Персональные
  рекомендации»): собирает шаблонную фразу вида "ваши последние N входов
  после импульсного роста по SYMBOL имеют отрицательный результат в X%
  случаев" — не LLM-генерация, соединение уже посчитанных чисел.

Фаза 3 (DNA Score, design → implementation по той же спецификации):
- `compute_dna_score()` — 4 компонента (`edge`/`discipline`/
  `behavior_stability`/`consistency`), веса зафиксированы константой
  `DNA_SCORE_WEIGHTS` (как `JudgeAgent.WEIGHTS` — не подбираются моделью).
  `score=None` с `confidence='insufficient_data'` при `total_trades < 20`
  (`DNA_SCORE_MIN_TRADES`) — тот же принцип честности, что `active=False` в
  `compute_dna_adjustment()`. `confidence='low'` при 20-49 сделках, `'full'`
  при 50+. Компонент `consistency` исключается (не штрафуется нейтральным
  значением), если нет минимум двух баков (символ/сессия) с ≥5 сделками —
  веса остальных компонентов перенормируются на оставшуюся сумму.

**UI**: новая кнопка "🧬 Trader DNA" в AI-меню (`core/keyboards.py`,
`ai_menu_keyboard()`) → `handlers/ai.py:show_trader_dna()` →
`ai/trader_dna.py:format_dna_report()`. Отдельно от "🎯 AI Coach"

Trader DNA прозрачный и детерминированный, коуч — LLM-интерпретация; оба
читают данные, но не дублируют друг друга (коуч не знает о
`behavior_events`, DNA v1 не вызывает LLM).

Проверено смоук-тестом с синтетической БД (30 сделок, поведенческие
события `fomo`, привязанные по времени к части входов по BTC-USDT):
`analyze_patterns()`/`compute_dna_score()`/`build_pattern_insights()`
отработали на реалистичных данных без ошибок; `build_pattern_insights()`

**Инцидент при тестировании (не попал в прод)**: первый прогон смоук-теста
по ошибке создавал `Database(db_path=...)` уже после того, как

тестовой БД. Оставлено здесь как напоминание на будущее: для
многобазовых тестов `Database()` всегда нужен явный сброс `_instance`, не

## Trader DNA v2: реальный баланс/dca_count/order_id (2026-07-09)

По итогам согласованного роадмапа (`TRADER_DNA_V1.md`, DNA v2) — три
точечных фикса существующей логики записи/чтения (не только чтение, как
v1). Каждый проверен смоук-тестом на изолированной БД (`Database._instance
= None` перед конструированием, см. предыдущий раздел) и мокнутым BingX.

**1. Реальный `balance` в `TradeScorer.score()`.** Раньше
`AIOrchestrator.review_closed_trade()` вызывал `TradeScorer.score(trade)`
вообще без `context` → компонент `risk_per_trade` всегда уходил в
нейтральный дефолт `5` (см. `TRADER_DNA_V1.md` §1.1). При расследовании
выяснилось, что причина глубже: `ContextBuilder.build_for_closed_trade()`
вообще не собирал `portfolio`/баланс (в отличие от
`build_for_open_position`/`build_for_new_setup`) — балансу неоткуда было
взяться, даже если бы вызывающий код догадался его передать. Исправлено:
- `ai/context_builder.py:build_for_closed_trade()` теперь тоже строит
  `portfolio` (тот же `_build_portfolio_context()`, что у двух других
  режимов) и включает его в контекст.
- `ai/consensus_engine.py:analyze_closed_trade()` считает
  `TradeScorer.score(trade, context={...})` один раз, с реальными
  `balance`/`losing_streak` из уже собранного контекста, и кладёт
  результат в `context['score']`.
- `_run_agents_parallel()` переиспользует `context['score']` вместо
  повторного слепого пересчёта для `trade_score`, который видит
  `JudgeAgent`; результат `_run_agents_parallel()` теперь включает
  `score_breakdown` в возвращаемый словарь.
- `AIOrchestrator.review_closed_trade()` берёт `score_breakdown` из
  результата консилиума вместо повторного независимого вызова
  `TradeScorer.score(trade)` без контекста (fallback без контекста
  остаётся только на случай `_error_response()`).
- Побочный эффект (не баг): `RiskAgent`/`RiskRuleEngine.assess()` для
  режима `post_trade` тоже получают реальный `portfolio` вместо `{}` —
  строго лучше, не меняет контракт (`ctx.get("portfolio", {})` уже был
  готов к любому значению).
- Проверено смоук-тестом (мокнутый BingX/агенты/Judge): `risk_per_trade`
  для сделки с реальным риском ~10% от депозита корректно посчитан как
  `2/10` вместо нейтральных `5/10`, воспроизведено и на уровне
  `ConsensusEngine.analyze_closed_trade()`, и на уровне
  `AIOrchestrator.review_closed_trade()`.

**2. `dca_count` доживает до `closed_trades`.** Раньше терялся при
закрытии сделки — жил только в `open_trades`, `_build_closed_trade()` не
включал это поле, а в `closed_trades` такой колонки не было вообще.
Исправлено: колонка `dca_count INTEGER DEFAULT 0` добавлена в
`closed_trades` (safe-миграция через `ALTER TABLE`, как и остальные),
`close_trade_atomic()`/`add_closed_trade()` сохраняют её,
`services/auto_sync.py:_build_closed_trade()` переносит значение из
`stored_open`. Добавлен `ai/trader_dna.py:dca_pattern()` — сравнение доли
убыточных исходов у сделок с усреднением vs без (данные будут расти
только для новых сделок, закрытых после этого изменения — историю не
восстановить).

**3. `order_id` в `behavior_events`.** Раньше джойн поведенческого флага к
конкретной сделке был только приближённым — по совпадению символа и
близости времени (окно 15 минут), потому что `behavior_events` не хранила
`order_id` вовсе (в отличие от `trade_events`, где он есть с Этапа 8).
Исправлено: колонка `order_id TEXT` + индекс добавлены в `behavior_events`
(safe-миграция), `BehaviorEngine.save_event()` принимает `order_id`,
`services/auto_sync.py` передаёт его при сохранении `revenge_trading`/
`fomo` (из `trade.get('orderId')` на открытии) и `panic_close` (из
`closed_trade.get('orderId')` на закрытии); `overtrading` намеренно без
`order_id` — событие не привязано к одной сделке.
`ai/trader_dna.py:behavior_symbol_pattern()` теперь джойнит точно по
`order_id`, когда он есть, и падает на старый приближённый джойн по
времени только для событий, записанных до этого изменения (или для
`overtrading`).

**Найдена и исправлена ошибка миграции при первой попытке.** Индекс
`idx_behavior_order_id` изначально был добавлен в тот же
`executescript()`, что и `CREATE TABLE IF NOT EXISTS` — для существующих
БД (где таблица уже была без `order_id`) это `no-op`, и `CREATE INDEX`
падал с `no such column: order_id`, потому что колонка добавляется позже,
в цикле `ALTER TABLE`. Поймано локальным прогоном на уже существующей
тестовой БД (не на новой) — если бы протестировали только на чистой БД,
баг остался бы незамеченным до продакшена. Исправлено: `CREATE INDEX
idx_behavior_order_id` перенесён после цикла `ALTER TABLE`, туда же, где
уже создаются `idx_open_orderId`/`idx_closed_orderId`. Проверено на обеих
БД — со старой схемой и с нуля.

**`setup_type` (Liquidity Sweep/FVG/BOS/CHOCH/Retest/Breakout/Scalp) — не
трогали.** Аудит: `services/structure_engine.py:analyze_structure()`
честно умеет отличать только тренд и HH/HL/LH/LL (BOS/CHOCH-подобные
сигналы), но не умеет определять FVG или Liquidity Sweep — для них в коде
просто нет детектора. Автотег в момент закрытия по structure на закрытии
(а не на входе) был бы к тому же неточным — структура на выходе ≠
структура на входе, а её никто не сохраняет на момент открытия. Решение
пользователя: вынести это отдельным пунктом, не смешивать с текущим
этапом DNA v2 (см. переписку) — остаётся ручным полем, как и раньше.

## Побочная находка: SL/TP не детектировались, если выставлены не *_MARKET ордером (2026-07-09)

Пользователь сообщил о позиции (INJ-USDT LONG), где секции Risk/Psychology
консилиума писали "стоп-лосс и тейк-профит не установлены", а сразу под
ними блок "Решение по позиции" показывал конкретные `Hard SL`/`TP1`/`TP2`
— выглядело как противоречие ("risk не видит стопы и тейки").

Расследование показало два независимых момента:

1. **Не баг, а нечёткая формулировка**: `Hard SL`/`TP1`/`TP2` в блоке
   "Решение по позиции" — это *расчётные* уровни от
   `services/stop_engine.py`/`services/tp_engine.py` (по
   поддержкам/сопротивлениям), а не факт того, что на бирже реально стоит
   ордер. Показывались всегда, независимо от реального состояния позиции.
   Исправлено: `utils/formatting.py:format_position_plan()` теперь
   помечает их как "Рекомендуемый Hard SL"/"Рекомендуемый TP1", и если
   реальных SL/TP на бирже нет — добавляет явную строку "⚠️ На бирже SL/TP
   не выставлены — выше только расчётные уровни". Флаги `_real_sl`/
   `_real_tp` прокидываются из `ai/orchestrator.py:build_position_plan()`
   (из уже нормализованной позиции), чтобы предупреждение показывалось
   только когда оно действительно уместно.

2. **Настоящий баг**: `services/bingx_api.py:get_open_positions()`
   реконструирует `stopLoss`/`takeProfit` позиции из списка открытых
   ордеров (`/openApi/swap/v2/trade/openOrders`), но проверял только типы
   `STOP_MARKET`/`TAKE_PROFIT_MARKET`. BingX поддерживает для стоп-лосса/
   тейк-профита два варианта триггера — market (`STOP_MARKET`/
   `TAKE_PROFIT_MARKET`) и limit (`STOP`/`TAKE_PROFIT`) — подтверждено по
   официальной документации BingX. Если ордер был выставлен как лимитный
   (частый случай для тейк-профита — исполняется точно по цене, без
   проскальзывания), код тихо считал, что SL/TP не установлены, хотя они
   реально были на бирже. Это, а не UI-формулировка, была вероятной
   причиной жалобы пользователя. Исправлено: типы `STOP`/`TAKE_PROFIT`
   добавлены в проверку рядом с `_MARKET`-вариантами.

Проверено: смоук-тестом с мокнутым `_request_with_retry` — отдельно для
лимитного варианта (`STOP`/`TAKE_PROFIT`, ранее не детектировался) и для
маркет-варианта (`STOP_MARKET`/`TAKE_PROFIT_MARKET`, регрессии нет).
Полный импорт `bot.py` — без ошибок.

## Устранение конфликта JudgeAgent vs ai_decision_engine — Вариант C (2026-07-10)

Полный аудит и три варианта архитектуры — в `DECISION_FLOW_AUDIT.md`.
Пользователь утвердил Вариант C: `ai_decision_engine` перестаёт быть
независимым вторым вердиктом и становится специализированным аналитиком
внутри `JudgeAgent`, единого источника решения.

**Новый модуль `ai/engines/structure_arbiter.py`** — единственное место,
где: строится `position_plan` (`build_structure_plan()`, перенесено сюда
из `AIOrchestrator.build_position_plan()` без изменения логики), решается,
является ли структурное решение объективным override-фактом
(`get_structure_override()`), считается структурный компонент скора для
остальных сигналов (`structure_score()`). Override ограничен ровно тем,
что `analyze_decision()` уже умел объективно определять — новых
детекторов не добавлялось:
- `EXIT` (пробой инвалидации, `stop_engine` статус `exit`) → форсирует `CLOSE`.
- `FULL_TP` (цена у TP2) → форсирует `CLOSE`.
- `PARTIAL_TP`/`DCA`/прочие статусы `stop_engine` — НЕ override, входят в
  `structure_score` как обычный компонент скора.

**`ConsensusEngine.analyze_open_position()`** строит `position_plan`
параллельно со сбором контекста (`asyncio.gather`) и передаёт в
`JudgeAgent.synthesize(position_plan=...)`. Результат `_run_agents_parallel()`
теперь включает `position_plan` — `AIOrchestrator.review_open_position()`
больше не пересчитывает его отдельным вызовом (устранены и риск
рассинхрона между двумя вызовами при движении цены между ними, и лишний
сетевой запрос).

**`JudgeAgent.synthesize()`**: новый параметр `position_plan`. Если
`structure_score()` не `None` (только `mode='open'` при доступных
данных) — используются веса `OPEN_WEIGHTS_WITH_STRUCTURE` (market 0.30 /
risk 0.30 / psychology 0.15 / trade 0.10 / structure 0.15) вместо базового
`WEIGHTS`; иначе поведение (post_trade/setup/нет данных) не меняется.
После расчёта `final_score` (с DNA-поправкой) `get_structure_override()`
форсирует `verdict` независимо от скора при срабатывании; `warnings`
получает explicit `"⚡ Принудительное решение: ..."`, `summary` для
EXIT/FULL_TP формулируется отдельно (не общим "закрыть", а с явной
причиной — сломанная идея vs достигнутая цель). Результат содержит
`structure_score`/`structure_override` — прозрачно, не скрыто внутри
скора.

**`utils/formatting.py:format_position_plan()`** — заголовок блока
показывает "Решение по позиции" только когда `plan.decision` совпадает с
override, который уже применён в вердикте Judge; иначе — "Структурная
деталь (не решение Judge)", чтобы блок не читался как второй,
конкурирующий вердикт.

**`core/scheduler.py:position_watch_job()`** вызывает ту же
`get_structure_override()`, что и `JudgeAgent` — при override шлёт
сообщение с явной пометкой "Тот же вердикт, что покажет /consilium"; без
override — информационный сигнал с пометкой "не финальное решение" и
отсылкой к `/consilium` за вердиктом, а не выдаёт структурный
HOLD/DCA/PARTIAL_TP за решение.

**Старый путь удалён, не оставлен рядом**: раздельное построение
`position_plan` в `AIOrchestrator.review_open_position()` убрано в рамках
этой же реализации (не отложено на будущий этап) — `AIOrchestrator.build_position_plan()`
остался только как тонкая обёртка над `build_structure_plan()` для
`position_watch_job`.

Проверено смоук-тестами (мокнутый BingX/агенты, изолированная БД):
- Воспроизведён репортнутый баг-кейс (высокое плечо, нет реального SL/TP,
  цена не пробила ни инвалидацию, ни TP) — подтверждено устранение:
  `⚖️ Вердикт: CLOSE` больше не сопровождается противоречащим
  `✅ Решение по позиции: HOLD`.
- Override `EXIT` форсирует `CLOSE` даже при `market_score=risk_score=
  psychology_score=80` (высокие LLM-оценки не могут перевесить объективный
  факт пробоя инвалидации).
- `get_structure_override()`/`structure_score()` — прямые unit-проверки
  таблицы override и edge-кейсов (`None`/`{}`).
- `build_structure_plan()` вызывается ровно 1 раз на запрос
  `review_open_position()` (не 2 — не дублируется между Judge и Orchestrator).
- `position_watch_job` — обе ветки (override/информационная) дают
  ожидаемый текст без слова "решение" вне override-случая.
- Полный импорт `bot.py`, регрессия по DNA v2 смоук-тестам — без ошибок.

## КРИТИЧЕСКИЙ БАГ: SL/TP терялись в One-Way Mode (2026-07-10)

Реальный кейс: SL/TP реально выставлены на бирже вручную, но
`get_open_positions()` возвращал `stopLoss: None, takeProfit: None`.
Полное расследование — статический анализ кода + документация BingX (нет
сетевого доступа к BingX/API-ключей пользователя в этой среде, живые raw
JSON не сняты, гипотеза подтверждена только на уровне кода+документации).

**Корень**: `services/bingx_api.py:get_open_positions()` сопоставляет
ордера из `/trade/openOrders` с позициями из `/user/positions` по ключу
`"SYMBOL_positionSide"`. Сторона ПОЗИЦИИ считается из знака `positionAmt`
(всегда `LONG`/`SHORT`), а `positionSide` ОРДЕРА бралось как есть из
ответа API. BingX документирует: **в One-Way Mode (сейчас default-режим
для новых аккаунтов) `positionSide` ордера может быть только `"BOTH"`**
— ключ ордера (`"SYMBOL_BOTH"`) никогда не совпадал с ключом позиции
(`"SYMBOL_LONG"`), поэтому реально существующий SL/TP терялся при
сопоставлении, хотя API его корректно возвращал. Не баг BingX API — баг
нашего парсинга (неявное предположение Hedge Mode).

Прошлый фикс (`STOP`/`TAKE_PROFIT` наравне с `_MARKET`-вариантами) был
необходим, но недостаточен — даже с ним сопоставление всё равно падало на
этом же шаге для One-Way Mode аккаунтов.

**Исправлено**: ключ сопоставления теперь зависит от `positionSide`
ордера — `"SYMBOL"` (без стороны), если `positionSide == 'BOTH'`
(One-Way Mode — на символ только одна позиция, неоднозначности нет), иначе
`"SYMBOL_positionSide"` как раньше (Hedge Mode — по символу+стороне, чтобы
не перепутать одновременные LONG и SHORT на одном символе). При
сопоставлении с позицией проверяются оба варианта ключа — режим аккаунта
заранее не известен, но в реальности совпадёт только один из них.

Проверено смоук-тестами (4 сценария, мокнутый `_request_with_retry`):
One-Way Mode (`positionSide="BOTH"`, воспроизведён репортнутый баг-кейс
INJ-USDT) — теперь корректно подтягивает SL/TP; Hedge Mode с одной
позицией — регрессии нет; Hedge Mode с ОДНОВРЕМЕННЫМИ LONG и SHORT на
одном символе — SL/TP не перепутались местами между сторонами; отсутствие
ордеров — SL/TP корректно остаются `None`. Полный импорт `bot.py` — без
ошибок.

**Не проверено на живых данных** — рекомендация пользователю: после
деплоя проверить `/consilium` по реальной позиции с вручную выставленным
SL/TP и подтвердить, что теперь отображается `Exchange SL`/`Exchange TP`
с реальными значениями, а не "отсутствует".

## SOL-USDT — фикс One-Way Mode не закрыл кейс, добавлена diagnostic-команда (2026-07-10)

Рекомендация из предыдущего раздела не подтвердилась: на реальной позиции
SOL-USDT с вручную выставленным "Т-п/с-л Вся позиция" (80.540/78.595 —
видно в приложении BingX) `/consilium` по-прежнему показывал `Exchange
SL: отсутствует`/`Exchange TP: отсутствует`. Оба предыдущих фикса (тип
ордера `STOP`/`TAKE_PROFIT`, ключ `positionSide` для One-Way Mode) были
основаны на официальной документации BingX и community-репортах
(ccxt-issues), не на реальном ответе API для этого конкретного аккаунта
— оба разумны сами по себе, но не гарантированно покрывают именно
"Whole position TP/SL" (`closePosition: true`).

Дальнейшее гадание по документации признано непродуктивным (третья
подряд спекулятивная правка без проверки на реальных данных — риск
переусложнить сопоставление ещё сильнее, так и не найдя настоящую
причину). Вместо этого добавлена **временная diagnostic-команда**
`/debug_positions` (`handlers/system.py:debug_positions_command`,
зарегистрирована в `bot.py`) — отправляет в Telegram сырой,
необработанный ответ `/openApi/swap/v2/user/positions` и
`/openApi/swap/v2/trade/openOrders` напрямую, в обход
`get_open_positions()`/маппинга SL/TP полностью. Не требует доступа к
серверным логам (бот работает на отдельном сервере, не в этой сессии) —
результат сразу виден в чате. Доступна только владельцу бота (тот же
`_check_chat()` gate, что у остальных admin-команд).

Проверено смоук-тестом с мокнутым `_request_with_retry`: команда
корректно отправляет обе raw-секции одним/несколькими сообщениями (через
`send_long`), возвращает пусто для чужого `chat_id`. Пойман и исправлен
баг при написании самой diagnostic-команды: изначально пытался
редактировать `update.message` (сообщение пользователя) вместо
собственного отправленного сообщения бота — `Message.edit_text()`
работает только на сообщениях, отправленных самим ботом; исправлено по
образцу остальных хендлеров (`msg = await update.message.reply_text(...);
await _send_long(msg, text)`, без `delete()`).

**Следующий шаг**: пользователь выполняет `/debug_positions` по SOL-USDT,
результат разбирается по существу (реальный `type`/`positionSide`/
`closePosition` для этого ордера), и только после этого — точечный фикс,
без дальнейших предположений. Команду убрать после закрытия кейса.

## КРИТИЧЕСКИЙ БАГ: цикл сопоставления TP/SL падал на КАЖДОМ синке (2026-07-10)

Пользователь получил логи сервера (`journalctl -u ai-helper-bot`) вместо
вывода `/debug_positions` — и это оказалось даже полезнее: логи показали
**реальную причину**, не гипотезу.

На каждом цикле синхронизации (каждые 15 секунд):
```
services.bingx_api - WARNING - get_open_positions: не удалось получить
TP/SL из openOrders: could not convert string to float: ''
```

**Корень**: `order.get('stopPrice', 0)` отдаёт дефолт `0` только если
ключа `stopPrice` нет вовсе. Если ключ есть, но значение — пустая строка
`''` (реальный случай минимум для одного из ордеров, похоже на "Т-п/с-л
Вся позиция" — `closePosition: true`), метод возвращает `''`, и
`float('')` кидает `ValueError`. Это исключение ловилось ВНЕШНИМ
`try/except` вокруг **всего** цикла сопоставления ордеров — при первом же
таком ордере обработка обрывалась целиком, и ни один ордер (включая
абсолютно нормальные, у других позиций) не успевал сопоставиться. Это
объясняет, почему НИ ОДИН из трёх предыдущих фиксов (тип ордера,
`positionSide`/One-Way Mode) не сработал — код падал раньше, чем
доходил до этой логики, **на каждом синке, для всех позиций**, а не
только для SOL-USDT.

**Исправлено**: `float(order.get('stopPrice', 0))` заменено на
try/except **вокруг разбора каждого отдельного ордера** — сбойный ордер
пропускается (с `logger.warning`, включающим полный сырой ордер для
дальнейшей диагностики), обработка остальных ордеров продолжается.

Проверено регрессией (5 сценариев, включая новый — пустой `stopPrice` у
одной позиции не должен мешать корректному сопоставлению у другой,
идущей в списке ордеров позже): One-Way Mode, Hedge Mode (одна позиция),
Hedge Mode (LONG+SHORT одновременно), отсутствие ордеров, пустой
`stopPrice` вперемешку с валидными ордерами — все 5/5 пройдены. Полный
импорт `bot.py` — без ошибок.

**Не полностью закрыто**: сам фикс восстанавливает TP/SL для позиций,
где `stopPrice` у ордера валиден (что почти наверняка было сломано для
ВСЕХ позиций из-за падения на каждом синке — значит этот фикс сам по
себе может быть значимым улучшением уже сейчас). Но для ордера с
изначально пустым `stopPrice` (похоже, именно "Вся позиция" TP/SL)
данные всё ещё не подтягиваются — сам факт пустого `stopPrice`
означает, что реальная цена триггера для такого ордера лежит в каком-то
ДРУГОМ поле, которое ещё предстоит определить. Новый `logger.warning`
теперь логирует полный сырой ордер при такой ошибке — следующая проверка
логов покажет точное поле.
