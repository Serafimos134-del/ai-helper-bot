# Trade Management Engine v3 — исследовательская ветка

Статус: **концепт, не реализован**. Рабочая модель сопровождения сделок —
v2 (описана ниже как "что уже есть"). Этот документ фиксирует видение v3,
сверенное с текущим состоянием кода, чтобы отдельные идеи можно было
забирать в v2 постепенно, без переписывания архитектуры целиком.

Источник: обсуждение с владельцем продукта, 2026-07-09, по итогам Этапов
1-6 плана AI Trading Core (см. `AUDIT.md`).

## Философия

> AI Helper Bot — это не бот сигналов. Это операционная система трейдера.

Не "покупай BTC", а "если ты решил торговать — я возьму управление сделкой
на себя". Это согласуется с направлением, в котором уже идёт `AIOrchestrator`
(единая точка входа, "единое аналитическое заключение") — v3 не меняет эту
философию, а углубляет её в сторону **сделки как объекта с состоянием**, а
не набора независимо считаемых величин (стоп, тейк, вход).

## Разделы v3 и их статус в v2

### 1. Trade Classification (CORE / STANDARD / HIGH_RISK)

Идея: перед открытием сделки AI относит актив к профилю (BTC/ETH — core,
большинство альтов — standard, мемкоины/новые листинги — high risk), и
профиль влияет на размер позиции, частоту обновления анализа, агрессивность
защиты прибыли.

**В v2 уже есть основа**: `utils/liquidation.py:get_volatility_class()` —
LOW (BTC/ETH) / MEDIUM (популярные альты) / HIGH (остальное). Используется
сейчас только для maintenance margin rate и SL/TP fallback %. Не используется
для частоты опроса или агрессивности DCA — это и есть конкретная дельта,
которую можно взять из v3 почти бесплатно.

### 2. Market Context

Тренд/структура/поддержка/сопротивление — уже есть: `services/structure_engine.py`,
используется в `stop_engine`/`tp_engine`/`ai_decision_engine` (Этап 4) и в
`trade_plan` для новых сетапов (Этап 6). Отклонение входа при неподходящем
рынке — уже частично реализовано через `JudgeAgent` (verdict AVOID/WAIT) и
`ConsensusEngine._is_market_data_valid()`.

### 3. Position Planning (вход, Hard Stop, DCA-план, TP-зоны, размер позиции)

Реализовано в Этапе 6 (`AIOrchestrator._build_setup_plan`) — почти дословно
то, что описано в v3, кроме явного "DCA-плана" (сейчас DCA не планируется
заранее для сетапа, только оценивается на живой позиции в `ai_decision_engine`).

Предложение "калькулятор перестаёт быть отдельной функцией, становится
частью RiskAgent" — организационное, не архитектурное; текущее разделение
`services/` (детерминированные движки) / `ai/agents/` (LLM-обёртки с
контекстом) сознательное и рабочее. Не переносить без конкретной причины.

### 4. Trade Lifecycle (NEW → BUILDING → CONFIRMED → RUNNER → EXIT)

**Единственная по-настоящему новая часть v3.** Всё остальное — это расчёты
(stateless: вход → число), а стадии — это **состояние**, которое нужно
хранить и переводить во времени:

- новая колонка в `open_trades` (`trade_stage`) + история переходов;
- правила перехода между стадиями (по силе движения, PnL%, структуре);
- на каждой стадии — разное поведение (Stage 1 NEW: контроль Hard Stop и
  DCA; Stage 3 CONFIRMED: Recommended Stop + первая частичная фиксация;
  Stage 4 RUNNER: структурный трейлинг).

Сегодня `services/ai_decision_engine.py:analyze_decision()` — точечный
снимок ("что делать сейчас"), без памяти о фазе жизни сделки. Добавление
стадийности — реальная новая сложность (миграция схемы + машина состояний),
не косметика. Именно поэтому это в исследовательской ветке, а не в
Этапе 7.

### 5. Stop Management (Hard Stop / Recommended Stop)

Полностью совпадает с уже реализованным в `services/stop_engine.py`
(Этап 4): `hard_sl` — где идея ломается, `recommended_sl` — как защитить
уже заработанное (`status`: keep/breakeven/profit_protection/trailing).
Философия v3 здесь ничего не меняет, только подтверждает текущий подход.

### 6. Target Zones вместо "Take Profit"

Терминологическое предложение — TP1/TP2/Runner уже по сути "зоны"
(структурные уровни, не фиксированные цели), просто называются TP1/TP2/TP3
в Этапах 4 и 6. Переименование дешёвое, можно сделать в любой момент без
архитектурных последствий — не приоритет.

### 7. Decision Engine (единый вердикт)

Уже близко к текущему `position_plan` (Этап 4: `decision`, `hard_sl`,
`recommended_sl`, `tp1/tp2/runner`, `reason`, `confidence`) и `trade_plan`
(Этап 6). v3 добавляет туда `Trade Stage` — единственное реальное дополнение
после того, как стадийность (п.4) будет реализована.

### 8. Consilium как единственная точка решений, общий Trade Context

Направление, в котором `AIOrchestrator` уже движется. Пункт "агенты больше
не считают стопы отдельно" не совсем соответствует текущему коду: `RiskAgent`
всё ещё считает свой `risk_score` независимо от `stop_engine`/`tp_engine` —
они решают разные задачи (риск-скор для Judge vs конкретные уровни для
пользователя), выносить их в один расчёт пока не требуется.

### 9. Минимальный /status, 4 команды пользователя

UX-упрощение, не архитектура. Не входит в v3 как техническая зависимость —
можно делать независимо в любой момент.

## Итог

Рабочей моделью остаётся v2. Единственная часть v3, которая требует
реального проектирования (новое состояние, миграция схемы, машина
состояний) — Trade Lifecycle (стадии сделки). Остальные пункты v3 либо уже
есть в v2 в близком виде, либо дешёвые точечные добавления (volatility-тир
для DCA/агрессивности стопа, переименование TP→Target Zones), которые можно
занести в v2 по мере необходимости без пересмотра архитектуры.

**Следующий шаг**: Этап 7 плана AI Trading Core (интеллектуальное
сопровождение сделки — перенос стопа, безубыток, частичная фиксация,
добор, полное закрытие) реализуется на v2, с учётом volatility-класса
актива (п.1) при выборе агрессивности рекомендаций. Стадийность (п.4)
остаётся в этом документе до отдельного решения о начале реализации v3.
