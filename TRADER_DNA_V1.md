# Trader DNA v1 — что можно извлечь из уже накопленных данных

Статус: **проектирование, код не писался, новые таблицы не создавались,
новые агенты не создавались**. Архитектура (`TraderContext`, интеграция в
`JudgeAgent`) уже построена и работает (см. `TRADER_INTELLIGENCE_ARCHITECTURE.md`).
Этот документ отвечает на другой вопрос: не "как объединить системы", а
"какой реальный интеллект можно извлечь из того, что эти системы уже
насобирали" — и честно отделяет то, что можно сделать прямо сейчас
чтением существующих данных, от того, что требует новых данных.

---

## Фаза 1: Аудит реальных данных

### 1.1 Что фактически лежит в каждом источнике

Аудит на уровне значений, а не только схемы — где данные реально
заполняются, а где колонка существует, но пуста или содержит заглушку.

| Источник | Поле/метод | Реальное состояние |
|---|---|---|
| `closed_trades` | `risk_percent` | **Всегда `0`** — `services/auto_sync.py:_build_closed_trade()` хардкодит `'risk_percent': 0` для каждой автоматически закрытой сделки. Колонка существует, но не несёт информации. |
| `closed_trades` | `risk_reward` | **Всегда `NULL`** — нигде не записывается после создания строки (`'risk_reward': None` в `_build_closed_trade()`, и никакой `update_trade_metrics()` вызов её не заполняет). `PerformanceEngine._risk_reward_analysis()` её тоже не читает — считает RR из `avg_win/avg_loss`, независимо. Колонка полностью мертва. |
| `ai/trade_scorer.py:TradeScorer.score()` | компонент `risk_per_trade` | **Всегда нейтральные 5/10** в живом пути. `AIOrchestrator.review_closed_trade()` вызывает `TradeScorer.score(trade)` без `context` — а без `context.get("balance")` метод сразу уходит в `else`-ветку с дефолтом. `score_breakdown.details.risk_per_trade`, который видит пользователь, ничего не измеряет на практике. |
| `open_trades` → `closed_trades` | `dca_count` | Живёт в `open_trades`, но **не переносится** в `closed_trades` при закрытии (`_build_closed_trade()` не включает это поле). История усреднений теряется в момент закрытия сделки. |
| `open_trades`/`closed_trades` | `stop_loss`/`take_profit` | Только **последнее известное значение** — `open_trades` перезаписывается каждый sync-цикл (15с), истории изменений нет. Если трейдер переносил стоп 3 раза, в БД останется только финальный. |
| `closed_trades` | `market_trend` | Заполняется, но **асинхронно постфактум** — `_build_closed_trade()` ставит `None`, значение приходит позже через `_analyze_and_notify()` → `db.update_trade_metrics(market_trend=...)`. Если фоновый анализ не успел/упал — остаётся `NULL`. |
| `closed_trades` | `setup_type` | Заполняется **только вручную** через кнопки `set_setup_*` в `core/router.py`. Для большинства сделок — `NULL`. Разброс по типам сетапа при малой выборке будет содержать 0-1 сделку на большинство категорий. |
| `behavior_events` | `metadata`/`order_id` | Событие содержит `symbol`, но **не `order_id`** — точный джойн "это событие произошло именно на этой сделке" невозможен, только приближённый (по символу и близости времени). |
| `trade_events` | весь механизм | Технически полностью готов (Этап 8, `order_id` есть) — но **накоплен совсем недавно**, объём на момент этого аудита близок к нулю. Механизм годный, данных для выводов пока физически мало. |
| RR — три независимых определения | `TradeScorer.rr_ratio` (по факту `|exit-entry| / |entry-SL|`) vs `PerformanceEngine.rr_ratio` (`avg_win / avg_loss`, по деньгам) vs мёртвая колонка `closed_trades.risk_reward` | Три разных числа с одним и тем же именем в разных местах интерфейса — не баг (каждое измеряет своё), но источник путаницы при построении паттернов; нужно явно указывать, о каком RR идёт речь. |

### 1.2 Какие паттерны можно выявлять уже сейчас

Без новых таблиц, без новых полей — только чтение того, что реально
заполняется:

| Паттерн | Источник данных | Как считается |
|---|---|---|
| Revenge trading / overtrading / FOMO / panic close | `behavior_events` (уже читается `TraderContext.behavior`) | Частота флагов за период, уже частично используется в advisory-поправке Judge |
| Ухудшение результата после серии убытков | `closed_trades.ai_score`/`score_breakdown`, упорядоченные по `close_time` | Сравнение `ai_score` сделок, идущих сразу после `current_loss_streak ≥ N`, с базовым средним |
| Результат по плечу (высокое плечо → хуже результат) | `closed_trades.leverage` + `realized_pnl` | Уже считает `PerformanceEngine._leverage_analysis()` (avg leverage в победах vs убытках) — не хватало только интерпретации как "паттерна", не новых данных |
| Лучшие/худшие символы | `closed_trades.symbol` + `realized_pnl` | `PerformanceEngine._setup_performance()` — готово |
| Лучшие/худшие сессии (время суток) | `closed_trades.close_time` | `PerformanceEngine._session_performance()` — готово |
| Результат по рыночному режиму (тренд/боковик) | `closed_trades.market_trend` (когда заполнен) | Не считается нигде сегодня, но это тривиальный `GROUP BY` по уже существующей колонке — новая функция, не новые данные |
| Переудержание позиций | `closed_trades.holding_minutes` | Сравнение длительности прибыльных vs убыточных сделок, либо против медианы по символу |
| Ранняя фиксация прибыли | `closed_trades.exit_price` vs `take_profit` (когда `take_profit` задан) | Доля прибыльных сделок, закрытых заметно ближе к `entry`, чем к `take_profit` |
| Слишком узкий/широкий стоп относительно собственной нормы | `closed_trades.stop_loss` vs `entry_price`, сравнение с собственным средним трейдера | Не абсолютная "правильность" стопа (внешней истины нет), а отклонение от **своей же** типичной дистанции |
| Калибровка Judge (вердикт на входе vs факт на выходе) | `trade_events.open_analysis` (JSON с `judge_verdict`) + `closed_trades.realized_pnl`, джойн по `order_id` | Механизм готов (Этап 8), но объём данных пока маленький (см. 1.1) — включать по мере накопления `trade_events` |

### 1.3 Какие паттерны требуют дополнительных данных

Честно — не пытаемся имитировать то, для чего данных физически нет:

| Паттерн | Почему нельзя сейчас | Что нужно (не в этом документе — только фиксация для v2) |
|---|---|---|
| Ранний вход (до подтверждения) | Нет сигнала "было ли подтверждение на момент входа" — сохраняется только факт входа, не контекст решения о таймингe | Отдельный сигнал на момент открытия (уже частично есть в `trade_events.open_analysis`, но не размечен как "рано/поздно") |
| Превышение риска (по деньгам) | `risk_percent` всегда `0`, `TradeScorer.risk_per_trade` всегда нейтрален (см. 1.1) | Передавать реальный `balance` в `TradeScorer.score()` при закрытии — это фикс существующего кода, не новая таблица; отдельное решение, не в рамках DNA v1 |
| Усреднение убыточных сделок (DCA в минус) | `dca_count` не сохраняется в `closed_trades` | Перенос `dca_count` из `open_trades` в `_build_closed_trade()` — снова фикс существующего кода, не новая таблица |
| Систематическое нарушение собственного плана (не эту рекомендацию проигнорировал, а паттерн в принципе) | Нет истории "рекомендация → действие трейдера" — `trade_events.companion` хранит только рекомендации, не то, что сделал трейдер в ответ | `position_adjusted`-события (уже описаны как гэп в `TRADER_INTELLIGENCE_ARCHITECTURE.md`, §1.4) — не реализовывались, остаётся гэпом и для v1 |
| Лучшие сетапы (по `setup_type`) | Заполняется только вручную, выборка на конкретный тип сетапа почти всегда < 5 | Автоподстановка `setup_type` из структуры рынка при закрытии — тоже был намечен как гэп, не реализован |
| Точный джойн поведенческого флага к конкретной сделке | `behavior_events` не хранит `order_id` | Добавить `order_id` в `behavior_events` — минимальное изменение схемы, сознательно не делается сейчас (см. "Важно" в начале) |

### 1.4 Статистическая устойчивость на 20 / 50 / 100 сделках

Не строгая статистика (для личного торгового бота избыточна), но
осознанные пороги — тот же принцип, что уже применён в `TraderContext`
(`MIN_TOTAL_TRADES=5`, `MIN_SYMBOL_TRADES=3`, поправка ограничена по
модулю). Разные паттерны требуют разного объёма, потому что часть из них
считается по **каждой** сделке (общие метрики), а часть — по **редким
подмножествам** (флаги поведения, конкретный символ, конкретный сетап).

| Объём | Что можно уверенно утверждать | Что всё ещё шум |
|---|---|---|
| **20 сделок** | Общий winrate/profit factor (весь пул), эффект плеча на исход (если плечо варьируется), эффект серии убытков на следующую сделку — метрики по каждой сделке в пуле | Разбивка по символам/сессиям/сетапам — типичное распределение даст 2-4 сделки на бакет, недостаточно; поведенческие флаги (revenge/fomo) — событие само по себе редкое, при 20 сделках может не встретиться ни разу |
| **50 сделок** | Символ-специфичные паттерны для 1-3 часто торгуемых символов (если торговля концентрирована, как обычно и бывает), сессионные паттерны, если сделки распределены по времени суток | Setup-type паттерны (при ручном, разреженном тегировании); поведенческие флаги всё ещё на грани — 2-4 срабатывания, разговор о "закономерности", а не единичном случае, только начинается |
| **100 сделок** | Поведенческие флаги как статистика (не единичные случаи), при активном использовании `set_setup_*` — начальные выводы по сетапам, калибровка Judge (open_analysis vs факт) уже на разумной выборке | Комбинации условий ("FOMO именно на BTC именно на высоком плече") — для пересечения двух-трёх редких условий нужно кратно больше |

Практическое следствие для реализации (не код, но принцип для Фазы 3):
**чем уже подмножество, тем выше порог минимума** — общие метрики можно
показывать с 5-10 сделок (как уже сделано в `TraderContext`), а
поведенческие/сетаповые паттерны разумно скрывать до 20-30 релевантных
случаев конкретно этого паттерна, а не общего числа сделок.

## Фаза 2: каталог паттернов Trader DNA v1

### Ошибки трейдера

| Ошибка | Доступно сейчас? | Источник |
|---|---|---|
| Поздний вход / погоня за движением (FOMO) | ✅ Да | `behavior_events` (`fomo`), приближённый джойн по символу/времени |
| Торговля после серии убытков (revenge trading) | ✅ Да | `behavior_events` (`revenge_trading`) + `streaks` в `TraderContext` |
| Овертрейдинг | ✅ Да | `behavior_events` (`overtrading`) |
| Ранняя фиксация прибыли | ✅ Частично (только там, где задан `take_profit`) | `closed_trades.exit_price` vs `take_profit` |
| Слишком узкий/широкий стоп (относительно своей же нормы) | ✅ Да | `closed_trades.stop_loss` vs `entry_price`, сравнение с собственным средним |
| Переудержание позиций | ✅ Да | `closed_trades.holding_minutes` |
| Ранний вход (до подтверждения) | ❌ Нет | Нужен сигнал на момент входа, которого не сохраняется |
| Превышение риска (в деньгах) | ❌ Нет — `risk_percent` всегда 0 | Требует фикса записи баланса при закрытии (вне рамок этого документа) |
| Усреднение убыточных сделок | ❌ Нет — `dca_count` теряется при закрытии | Требует переноса поля, не новой таблицы |

### Сильные стороны

| Сильная сторона | Доступно сейчас? | Источник |
|---|---|---|
| Лучшие символы | ✅ Да | `PerformanceEngine._setup_performance()` |
| Лучшие сессии (время суток) | ✅ Да | `PerformanceEngine._session_performance()` |
| Лучшие рыночные условия (тренд/боковик) | ✅ Да, но не считается сегодня — нужна новая функция чтения (не новые данные) | `closed_trades.market_trend` |
| Дисциплина по SL/TP (доля сделок с обоими выставленными) | ✅ Да | `closed_trades.stop_loss`/`take_profit` |
| Лучшие сетапы | ⚠️ Частично — данные есть, но разрежены (ручное тегирование) | `closed_trades.setup_type` |
| Лучшие сценарии сопровождения | ⚠️ Механизм готов, объём данных пока мал | `trade_events` (companion-цепочки) |

### Поведенческие закономерности

| Закономерность | Доступно сейчас? | Источник |
|---|---|---|
| Качество сделок падает после серии убытков | ✅ Да | `closed_trades.ai_score` в хронологии + `streaks` |
| Результат хуже на высоком плече | ✅ Да | `PerformanceEngine._leverage_analysis()` |
| FOMO-входы коррелируют с худшим исходом | ✅ Да, приближённо | `behavior_events` + `closed_trades`, джойн по символу/времени |
| Переудержание убыточных позиций | ✅ Да | `closed_trades.holding_minutes` для убыточных vs прибыльных |
| Систематическое нарушение собственных правил (не разовый случай) | ❌ Нет | Нужна история "рекомендация → действие" |

### Персональные рекомендации

Пример из ТЗ ("BTC Long выглядит хорошо, однако ваши последние 12 входов
после импульсного роста имеют отрицательный результат") — это не новая
категория данных, а **соединение уже доступных паттернов** в одно
предложение:

1. Текущий сетап помечен как потенциальный FOMO/impulse-вход (это уже
   умеет делать `MarketAgent`/контекст сделки на этапе анализа — резкое
   движение цены перед входом видно в `ticker.change_24h`/структуре).
2. Исторический паттерн "FOMO-входы по этому символу" (см. таблицу выше)
   даёт конкретную цифру — сколько таких входов было и какой у них исход.
3. Итоговое предложение — шаблонная фраза, собранная из (1) и (2), без
   новой LLM-генерации: "{market_analysis}, однако ваши последние
   {N} входов после импульсного роста по {symbol} имеют {результат}."

Технически это **новая функция чтения** (например
`ai/trader_context.py:build_pattern_insights(db, symbol, behavior_type)`),
не новая таблица и не новый агент — берёт `behavior_events` (по символу,
без точного джойна — по приближению) и соответствующие `closed_trades`,
считает долю негативных исходов, возвращает готовую фразу или `None`,
если данных недостаточно (см. Фаза 1, §1.4).

## Фаза 3: DNA Score — проектирование (не реализация)

### Компоненты

Тот же принцип прозрачности, что уже в `JudgeAgent.WEIGHTS`/
`TradeScorer.WEIGHTS` — явные веса, детерминированный расчёт, без ML:

| Компонент | Что измеряет | Источник |
|---|---|---|
| **Edge** | Базовая прибыльность (winrate/profit factor относительно безубыточности) | `PerformanceEngine._basic_metrics()` |
| **Discipline** | Доля сделок с выставленными SL и TP, дистанция стопа относительно собственной нормы | `closed_trades.stop_loss`/`take_profit` |
| **Behavior stability** | Обратная частота поведенческих флагов на N сделок (меньше флагов = выше стабильность) | `behavior_events` |
| **Consistency** | Насколько равномерен результат по символам/сессиям (не "звезда на одном инструменте, провал на всех остальных") | `PerformanceEngine._setup_performance()`/`_session_performance()`, дисперсия winrate между бакетами |

### Как считается

Взвешенная сумма (веса — предмет калибровки при реализации, не
фиксируются в этом документе) → шкала 0-100, той же ориентации, что и
`ScoringEngine`/`JudgeAgent` (выше = лучше). Считается по требованию (как
`PerformanceEngine.get_full_report()` сегодня), не персистируется —
тот же принцип, что и у `TraderContext`.

### Минимальные объёмы данных

Не единый порог на всё — по аналогии с §1.4:

- **Сам DNA Score не показывается** при `total_trades < 20` — вместо числа
  явный статус "недостаточно данных" (тот же принцип, что `active=False`
  в `compute_dna_adjustment()` сегодня — не выдумывать точность, которой
  нет).
- При `20 ≤ total_trades < 50` — показывается с явной пометкой низкой
  уверенности (например, `confidence: "low"` рядом со значением).
- **Компонент Consistency** требует отдельного порога — как минимум 2
  бакета (символа/сессии) с ≥5 сделками каждый, иначе не включается в
  сумму вовсе (не штрафуется отсутствием данных, а исключается).
- **Compound-паттерны** (пересечение двух условий, например "FOMO +
  высокое плечо") не участвуют в DNA Score v1 вообще — см. вывод §1.4 о
  том, что для них нужен кратно больший объём, чем для базовых
  компонентов.

### Как избежать ложных выводов на маленькой выборке

Три механизма, все уже опробованы в `TraderContext.compute_dna_adjustment()`
и переносятся на DNA Score без изменения философии:

1. **Явный порог активации** — компонент либо участвует в расчёте, либо
   явно помечен как "недоступен", никогда не подставляется нейтральным
   значением молча (тот самый баг с `TradeScorer.risk_per_trade`,
   найденный в §1.1, — урок именно про это: тихий нейтральный дефолт
   выглядит как настоящее число).
2. **Confidence вместе с числом, а не отдельно** — DNA Score всегда
   возвращается вместе с объёмом выборки, на основе которого он посчитан,
   так же как `dna_adjustment` сегодня возвращает `reason`, а не голое
   число.
3. **Никаких скрытых весов "на глаз"** — веса компонентов фиксируются в
   коде как константы (как `JudgeAgent.WEIGHTS`), не подбираются
   динамически моделью — иначе теряется предсказуемость, которая была
   осознанным архитектурным выбором для всего `AI Trading Core` (Judge —
   детерминированный арбитр, не LLM).

## Фаза 4: Roadmap Trader DNA

### DNA v1 (этот документ)

- **Новые возможности**: каталог паттернов из Фазы 2 (то, что помечено
  ✅), DNA Score (Фаза 3) как read-only метрика, персональные
  рекомендации, соединяющие анализ сетапа с историческим паттерном по
  символу.
- **Требования к данным**: только то, что уже собирается. Единственное
  новое — read-side функции (`build_pattern_insights()` и подобные), без
  изменения схемы.
- **Риски**: приближённый джойн `behavior_events` ↔ `closed_trades` (нет
  `order_id`) может давать неточные корреляции при высокой частоте сделок
  по одному символу в короткий период — два разных события за один день
  на одном символе можно перепутать. Смягчение: джойн по ближайшему
  времени с окном в несколько минут, при неоднозначности — не включать
  событие в статистику вместо угадывания.
- **Ожидаемый эффект**: более конкретные (не общие) персональные
  рекомендации в ответах консилиума; более точная advisory-поправка Judge
  (сейчас использует только `symbol_context`/`streaks`/`flag_counts`, без
  паттернов вроде "поздний вход конкретно на этом символе").

### DNA v2

- **Новые возможности**: точные паттерны нарушения собственного плана
  (не только "стоп был узким", а "трейдер сам подвинул стоп ближе, чем
  рекомендовал `position_watch_job`"); реальная оценка превышения риска;
  учёт истории усреднений; плотные данные по `setup_type`.
- **Требования к данным**: `position_adjusted`-события в `trade_events`
  (диф стоп/тейк/DCA при sync — уже намечено в
  `TRADER_INTELLIGENCE_ARCHITECTURE.md`, не реализовано); `order_id` в
  `behavior_events`; фикс записи реального `balance` в `TradeScorer`;
  автоподстановка `setup_type` из структуры рынка при закрытии, если не
  задан вручную. Все — точечные изменения существующих таблиц/вызовов,
  не новая архитектура.
- **Риски**: `position_adjusted`-события увеличат частоту записи в БД
  (тот же риск, что уже учтён и смягчён в Этапе 7 — писать только на
  реальное изменение значения, не на каждый sync-тик).
- **Ожидаемый эффект**: DNA Score перестаёт зависеть от приближённых
  джойнов; появляется реальный компонент "соблюдение плана", а не только
  прокси через дисциплину по SL/TP.

### DNA v3

- **Новые возможности**: хранимые закономерности (`trader_patterns`,
  спроектированы, но сознательно не создавались ни в `TRADER_DNA_ARCHITECTURE.md`,
  ни здесь) — не пересчитывать статистику каждый раз, а накапливать
  подтверждённые выводы во времени, с возможностью деактивации устаревших;
  качественная (не шаблонная) интерпретация паттернов через LLM поверх уже
  посчитанных детерминированных данных — здесь, и только здесь, может
  оказаться оправданным Вариант A из `TRADER_INTELLIGENCE_ARCHITECTURE.md`
  (отдельный агент), потому что задача перестаёт быть "агрегировать
  цифры" и становится "объяснить нетривиальную комбинацию паттернов
  словами" — то, для чего рассуждение LLM действительно даёт что-то
  сверх детерминированного шаблона.
- **Требования к данным**: устойчивый объём сделок (по опыту Фазы 1 —
  сотни, не десятки), накопленная история v2-полей достаточной глубины.
- **Риски**: главный риск версии — соблазн завести LLM-агента раньше, чем
  для этого появится реальный материал; без выполненного v2 агент будет
  рассуждать поверх тех же шаблонных данных, что и v1, только дороже и
  менее предсказуемо.
- **Ожидаемый эффект**: закономерности не пересчитываются с нуля на
  каждый запрос (быстрее, стабильнее между запросами); качественные
  объяснения там, где детерминированный шаблон недостаточно выразителен
  (например, объяснение *почему* паттерн существует, а не только что он
  есть).

---

## Итог

Решение о реализации DNA v1 — на усмотрение владельца продукта. Ключевой
для этого решения факт из аудита: часть данных, которые выглядят как уже
существующие (`risk_percent`, `risk_reward`, `TradeScorer.risk_per_trade`),
на практике либо всегда нулевые, либо нейтральные — DNA v1 должен строиться
на подтверждённо живых источниках (`behavior_events`, `PerformanceEngine`,
`closed_trades.ai_score`/`market_trend`/`holding_minutes`), не на тех,
что выглядят как данные, но данными не являются.
